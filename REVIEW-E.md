# REVIEW-E

## Verdict: CHANGES REQUIRED

## Findings

1. [MAJOR] src/mindroom/workers/backends/kubernetes.py:157-177 - The round-2 reporter decides whether to emit `cold_start`/`ready` using only the last `on_poll_tick()` value, but `wait_for_ready()` only calls that hook before each 1-second sleep. Any pod that crosses the 1.5-second grace window and becomes ready before the next poll therefore completes silently: `latest_elapsed` is still 1.0, `finalize("ready", None)` emits nothing, and the user sees no warmup notice even though startup exceeded the grace period. The new grace-window test misses this because it injects synthetic `on_poll_tick(1.6)` values that `wait_for_ready()` never produces on this path. Fix: capture one final monotonic elapsed value in `finalize()` (or make `wait_for_ready()` invoke `on_poll_tick()` immediately before returning ready) and add a regression test for the 1.5-2.0 second boundary case.
2. [MAJOR] src/mindroom/workers/backends/kubernetes.py:243-278 - Progress fan-out still breaks the planned same-worker label coalescing when a second caller joins after the first `cold_start` event. Newly registered sinks only receive future events, so if the worker becomes ready before the next 5-second waiting tick the late joiner gets only `ready`, `apply_worker_progress_event()` drops it because no warmup state exists yet, and that tool label never appears in the shared suffix. The added concurrent-waiter test only covers the favorable ordering where caller 2 subscribes before the first progress tick, so it misses this real path. Fix: store the current in-flight progress snapshot per `worker_key` and replay it to newly registered sinks before they wait for subsequent events, with a regression test that registers caller 2 after the initial `cold_start`.
3. [MAJOR] tests/test_workloop_thread_scope.py:87-148 - This ISSUE-183 branch adds unrelated workloop-fixture surgery and gates the whole regression file on a stale `types.py` checkout shape, but the current plugin checkout uses `runtime.py` instead (`~/.mindroom-chat/plugins/workloop/runtime.py`, no `types.py`, per ISSUE-180). In this environment that means `_plugin_checkout_available()` returns false and the entire workloop regression file now self-skips, so the branch silently drops real coverage in an unrelated area. Fix: drop this workloop hunk from ISSUE-183; if it must stay, update both the availability guard and the fixture rewrite block to target the current `runtime.py`-based layout instead of `types.py`.
